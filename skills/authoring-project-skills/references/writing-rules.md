# Writing Rules

Read this file when writing the actual rule/principle bullets inside a SKILL.md, rules/, or references/.

## 原则 + 检验句,而不是原则 + 解释

每条规则后面跟一个 Agent 执行后能**自我验证**的具体检验,而不是抽象声明。

```
# ❌ 声明 —— Agent 生成前只能抽象"记住",生成后没有钩子触发验证
保持简洁。写解决问题所需的最少代码。

# ✅ 原则 + 检验句 —— Agent 能在执行后真的去问 / 去跑
精准修改。只改必须改的,只清理你自己造成的乱。
✓Check:每一行改动都能直接追溯到用户的请求吗?
```

格式:
```
## 原则名称
一句话描述原则。
✓Check:[一个可以跑的命令 / 一个可以问自己的具体问题]
```

## 约束 + 上下文,而不是把每步写死

提供约束和"为什么",不要硬编码具体值 —— 过度指令化会让 skill 在依赖升级后整片失效。

```
# ❌ 过度指令化:换一套 token 命名,几百条硬编码全要改
添加按钮时使用 class "bg-blue-500 hover:bg-blue-700 ..."

# ✅ 约束 + 上下文:能多活过几次重构
按钮使用项目设计系统 token(见 rules/frontend-rules.md)。
交互元素必须有可见的 hover / focus 状态。
```

## 不要陈述显而易见的事

只写**项目特有约定、与主流不同之处、Agent 默认会出错的场景**。
通用知识(如"SQL 注入是坏事")模型已经知道,写了只是浪费 token。

判断标准:"资深开发者第一次看你的项目,什么会让他踩坑?" 那个才值得写。

## 用例子提醒最容易踩的反模式

通用的"别写烂代码"拦不住 Agent —— 它不知道什么算烂。用**真实的代码级 before/after**,
展示的不是明显错误,而是"看起来合理但时机错了"的改动(例如修 bug 的 PR 里顺手加校验 / docstring)。

## 一个好例子胜过多个平庸例子

- 选最相关的一种语言写**一个**完整、可运行、注释解释 WHY 的真实例子。
- 不要在 5 种语言里各写一遍,不要写填空模板,不要写造作的示例。

## 反模式
- 叙事式例子("在 2025-10-03 的会话里我们发现…")—— 太具体、不可复用。
- 多语言稀释 —— 质量平庸 + 维护负担。
- 把代码写进流程图节点 —— 不能复制、难读。
- 无语义的标签(step1 / helper2)。
