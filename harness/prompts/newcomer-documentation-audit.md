# 新人接手项目文档审查提示词

## 用途

当需要站在“新成员第一次接手项目”的视角，审查一个项目的文档是否完整、清楚、可信时，使用本提示词。

适合场景：

- 接手陌生项目之前做文档体检。
- 梳理 README、AGENTS、docs、架构文档、契约、运行说明和历史计划之间的冲突。
- 判断哪些历史 implementation plan / review / todo 应归档，哪些入口文档应补齐。

## 提示词正文

```text
我需要你站在“新成员第一次接手这个项目”的视角，系统审查项目文档是否足够完整、清楚、可信。

请先阅读项目入口文档和文档规范，例如 AGENTS.md、README、docs/README、docs/文档规范、架构索引、契约索引、运行/部署/测试说明。然后结合代码、配置、脚本和测试做交叉核对，不要只看文档表面。

重点检查：
1. 新人能不能顺利理解项目是什么、怎么启动、怎么测试、怎么部署、怎么排查问题。
2. 当前事实源是否清楚区分：架构事实、契约、实现计划、历史归档、review 记录、todo/backlog。
3. 是否存在过期、冲突、误导性文档，尤其是已经落地到代码里的设计但文档仍写旧方案。
4. 是否有关键入口缺失，例如模块职责、数据流、API 契约、存储边界、目录归属、运行依赖、验证命令。
5. 是否有历史 implementation plan、review、todo 长期停留在活动入口，导致新人误以为它们仍是当前任务。
6. 是否有“平台负责”“统一处理”“自动同步”“支持”等模糊说法，但没有说明 owner、代码路径、数据结构、失败处理或验证方式。

输出请按优先级给出：
- P0：会直接误导新人、导致错误实现或错误运维的文档问题。
- P1：新人能读懂但需要大量猜测、跳转或查代码才能确认的问题。
- P2：结构、索引、归档、措辞和可维护性问题。

每条问题请包含：
- 问题位置：文件路径或目录。
- 现状：现在文档怎么写。
- 风险：新人会怎么误解。
- 建议：应该补什么、删什么、迁到哪里或改成什么口径。
- 事实依据：对应代码、配置、测试、脚本或更权威文档。

最后给一个最小整改计划：哪些先改，哪些可以后续处理，哪些只需要归档或更新链接。
```

## 可选补充

如果已经知道某个项目存在具体风险，可以把下面这句追加到提示词末尾：

```text
另外请特别检查历史 impl-plan 是否应该归档，以及题包/资源/附件这类入口是否已经和当前存储适配器、运行时 owner、导入导出流程对齐。
```

## 使用要求

- 审查前先读目标项目自己的 `AGENTS.md` 和文档规范。
- 如果项目已有更强的文档归属规则，以项目规则为准。
- 不要把 implementation plan、review 或 todo 当成当前架构事实源。
- 输出建议应能对应到文件路径、代码路径、脚本或测试依据。
